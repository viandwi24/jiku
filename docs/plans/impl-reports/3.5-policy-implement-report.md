## Ringkasan

Plan 3.5 merevisi policy/ACL system dari Plan 3 yang terlalu rigid ‚Äî enum-based resource/subject types, policy langsung FK ke agent, rules di-load per request. Hasil revisi: policy sebagai reusable entity, pluggable subject matcher, kondisi evaluasi, dan runtime caching yang benar.

---

## Motivasi Revisi

| Problem di Plan 3 | Solusi di Plan 3.5 |
|-------------------|--------------------|
| `resource_type`/`subject_type` hardcoded enum | Open string ‚Äî developer define sendiri |
| Policy rules FK langsung ke `agent_id` | `policies` table tersendiri + `agent_policies` join |
| `checkAccess()` tidak pluggable | `SubjectMatcher` ‚Äî injectable custom matching logic |
| Rules di-load per request (`updateRules()` tiap chat) | Rules di-cache di runtime, di-sync hanya saat policy berubah |
| Runtime tidak permanent per project | `wakeUp/sleep` pattern ‚Äî runtime hidup selama project ada |

## Perbandingan Arsitektur

### Policy Rules ‚Äî Sebelum vs Sesudah

```
SEBELUM (Plan 3):

             (agent_id FK)

SESUDAH (Plan 3.5):
```

### Runtime Rules Loading ‚Äî Sebelum vs Sesudah

```
SEBELUM (Plan 3):
tiap chat request:
  resolveCaller()
  getPolicyRules(agentId)   ‚Üê DB query tiap request
  runtime.updateRules(rules) ‚Üê shared state mutation tiap request

SESUDAH (Plan 3.5):
saat wakeUp():
  loadProjectPolicyRules(projectId)  ‚Üê sekali saja

saat policy berubah (admin action):
  syncRules(projectId)  ‚Üê explicit reload

tiap chat request:
  resolveCaller()  ‚Üê satu-satunya DB query
  runtime.rules sudah ada di memory
```
